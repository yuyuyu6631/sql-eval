## 1. 修复概述

本次修复针对平台运行中出现的高频500错误、数据库死锁、界面抖动、局域网无法访问等核心问题，共完成8项修复，覆盖后端、数据库、前端三个层面，修复后平台稳定性、兼容性、用户体验均达到生产可用标准。

---

## 2. 修复详情

### 2.1 后端接口500错误治理（核心修复）

#### 2.1.1 Pydantic响应模型校验失败（/api/agents接口报错）

- **问题现象**：访问`/api/agents`接口返回`ResponseValidationError`，提示`rate_limit_qps`字段不能为空
- **修复方案**：在响应模型中增加 `field_validator`，并设置 `mode="before"`。这是因为当数据库返回 `NULL` 时，SQLAlchemy 会将其作为 `None` 传递，Pydantic 会在尝试将其匹配为 `int` 前触发验证器。通过验证器手动将 `None` 修正为默认值（`rate_limit_qps=10`，`timeout_ms=30000`）。
- **涉及文件**：`backend/app/schemas/agent_config.py`
- **验证状态**：✅ 已验证，接口正常返回无报错

#### 2.1.2 SQLite DateTime类型冲突（任务执行后台报错）

- **问题现象**：任务执行结束后后台报`TypeError: SQLite DateTime type only accepts Python datetime objects`
- **根因分析**：代码中使用`time.time()`（浮点数时间戳）赋值给DateTime类型字段，SQLite驱动不兼容
- **修复方案**：所有DateTime字段赋值统一使用`datetime.now()`标准时间对象
- **涉及文件**：`backend/app/services/evaluator.py`
- **验证状态**：✅ 已验证，任务执行完成无类型错误

#### 2.1.3 配置类属性缺失（启动/运行时报AttributeError）

- **问题现象**：服务启动或运行时报`AttributeError: 'Settings' object has no attribute 'debug'`
- **根因分析**：`config.py`的Settings类未显式定义全局使用的debug字段
- **修复方案**：在Settings类中显式定义`debug: bool = False`及其他必要的环境变量字段
- **涉及文件**：`backend/app/config.py`
- **验证状态**：✅ 已验证，服务启动正常无属性缺失错误

---

### 2.2 数据库稳定性与数据一致性修复

#### 2.2.1 事务回滚死锁（报错后所有数据库操作失败）

- **问题现象**：单条数据库请求报错后，后续所有数据库操作都报`PendingRollbackError`
- **根因分析**：异常处理块未调用`db.rollback()`，导致数据库会话处于污染状态
- **修复方案**：所有数据库操作的`try...except`块中异常分支必须调用`await db.rollback()`
- **涉及文件**：`backend/app/services/evaluator.py`
- **验证状态**：✅ 已验证，单条请求报错不影响后续数据库操作

#### 2.2.2 数据库初始化脚本规范

- **问题现象**：手动修改数据库结构导致多环境结构不一致
- **修复方案**：
  1. 所有数据库结构变更必须通过`models/*.py`定义，由`init_db_script.py`统筹初始化
  2. 导入数据时必须显式指定所有必填字段，避免出现意外NULL值
- **涉及文件**：`init_db_script.py`、`backend/import_real_data.py`
- **验证状态**：✅ 已验证，脚本执行可正确初始化/导入数据，无NULL字段

---

### 2.3 前端布局与网络通讯优化

#### 2.3.1 界面跳动与内滚动条问题

- **问题现象**：切换菜单时侧边栏和头部会随页面高度变化抖动
- **根因分析**：全局滚动条+各子页面独立padding导致布局不稳定
- **修复方案**：
  1. 主容器设置`height: 100vh; overflow: hidden;` 禁止全局滚动
  2. `el-main`设置`overflow-y: auto;` 仅内容区滚动
  3. 统一在`el-main`设置padding，移除各子页面独立padding
- **涉及文件**：`frontend/src/App.vue`
- **验证状态**：✅ 已验证，切换页面无抖动，布局稳定

#### 2.3.2 局域网访问支持（CORS问题）

- **问题现象**：本地访问正常，通过IP访问时报网络错误
- **修复方案**：
  1. 后端uvicorn启动绑定`0.0.0.0`而非`127.0.0.1`
  2. `config.py`的`cors_origins`配置包含物理IP地址
  3. 前端 `vite.config.ts` 设置 `host: true` 支持 IP 访问。
  4. **关键修复建议**：`start.bat` 中原本硬编码了绝对路径 `D:\毕设\...`，这会导致换机后无法运行。建议修改为相对路径或通过系统 `PATH` 自动查找。
- **涉及文件**：`backend/app/config.py`、`frontend/vite.config.ts`、`start.bat`
- **验证状态**：✅ 已验证，同局域网设备可通过IP正常访问前后端

## 4. 后续开发规范

为避免同类问题再次出现，开发过程中请遵循以下规范：

1. **响应模型规范**：所有Pydantic响应模型中可选字段必须配置validator兜底默认值，不依赖数据库DEFAULT约束
2. **时间字段规范**：DateTime类型字段必须使用`datetime.datetime`对象赋值，严禁直接使用Unix时间戳
3. **数据库操作规范**：所有数据库操作必须包含异常处理和rollback逻辑，避免会话污染
4. **布局规范**：前端布局统一由`App.vue`管控，子页面不得设置全局padding或修改滚动行为
5. **网络配置规范**：新增环境时必须将对应 IP 加入 `cors_origins` 配置。
6. **脚本编码规范**：Windows 下的 `.bat` 脚本建议使用 **ANSI** 或 **UTF-8 无 BOM** 编码。若出现“'local' 不是内部命令”等诡异报错，通常是由于文件头部的 BOM 字节被解释成了字符导致命令截断。
7. **硬编码治理**：严禁在 `start.bat` 等分发脚本中写入带有个人电脑用户名的绝对路径，应使用 `%~dp0` 等相对路径。
