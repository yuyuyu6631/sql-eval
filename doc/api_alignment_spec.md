# 前后端对齐规范 (API & Alignment Spec - UI/UX Pro Max 版)

在使用现代 UI 框架进行高质量交付（UI/UX Pro Max 要求）时，前后端数据通信的时效性、连贯性和错误响应精度更为重要。本规范在基础 RESTful 之上，增补了基于用户反馈与无障碍考量的接口应对标准。

## 1. 资源状态的即时反馈机制
- **网络防抖 (Debounce / Throttle)**: 
  - 前端在高频调用接口（如跨输入搜索等）必须强制防抖；后端也应提供一定限度的限流 (Rate Limiting) 防护。
- **精准的异常传递 (`error-feedback`)**:
  - UI/UX 规范强烈要求“明确的错误反馈”。后端在返回 4xx/5xx 时，`detail` 字段或报错结构不应包含难以读懂的服务器堆栈信息，而必须直接提供：
    1. 用户友好的展示文案（如 `目标记录不存在`）。
    2. 如果有字段级校验错误（如 FastAPI Unprocessable Entity），需带回具体的哪个字段产生了何种错误并让前端标注在**输入框的近距离旁**。

## 2. 长耗时与防阻塞对齐 (`loading-states` & `content-jumping`)
- 评估任务 (Evaluation Task) 等长耗时的接口：
  - 后端返回状态如 `Status: RUNNING`。
  - 结合 UI 规范中的 `loading-buttons` 和 `content-jumping` 禁止跳转，前端必须保留对应区块的 Skeleton (骨架屏) 继续轮询，禁止整个页面发生重绘 (Rerender)。后端需要保证轮询状态获取接口的毫秒级高可用响应。
  
## 3. 标准分页与数据加载机制
- UI 交互严禁无限制的长列表下拉。涉及表格或可导航列表的数据：
  - GET 列表数据必须提供完整的 `{ data, total, skip, limit }` 数据包装体。
  - 前端在执行“翻页”、“拉取更多”时，需将对应列表区域置于加载覆盖层（避免破坏交互连贯性）。

## 4. 字段规范与缓存对齐
- **数据字段的规范**: 
  - 无论前端采取 `camelCase` 或保留 `snake_case`，一经确定，禁止项目中英文混杂、风格混写。
  - 当带有状态枚举值（如 Status enum）时，前后端必须枚举对齐。对于状态不同的色彩响应（如 绿色表示通过、红色表示异常），请后端传“真实业务语义”而绝对不可以给前端传具体的“颜色值或UI类名”。
- **性能响应诉求 (Performance HIGH)**: 
  - 支持 HTTP Cache 机制以满足尽可能快的响应加载；对于静态资源和长效不变更的列表接口加上可合理的 ETags 等浏览器控制机制。
